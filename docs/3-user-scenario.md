## 1. 문서 개요

### 1.1 목적

본 문서는 TodoListApp을 실제로 사용하는 페르소나의 행동 흐름을 시나리오 형식으로 기술한다. 개발팀, 디자이너, QA가 사용자 관점의 기능 흐름을 공통된 기준으로 이해하고 검증할 수 있도록 한다.

### 1.2 시나리오 구성 원칙

- 페르소나 기반 실제 사용 맥락을 스토리 형식으로 서술
- 각 시나리오는 **목표 → 사전 조건 → 단계별 행동(사용자) + 시스템 반응 → 결과** 구조로 구성
- 정상 시나리오와 예외 시나리오를 모두 포함
- 각 시나리오에 관련 FR ID, BR ID를 명시적으로 연계

### 1.3 시나리오 ID 체계

| 접두사 | 도메인 | 예시 |
|--------|--------|------|
| SC-A | 인증 (Authentication) | SC-A-01, SC-A-02 |
| SC-U | 사용자 관리 (User) | SC-U-01 |
| SC-T | 할일 관리 (Todo) | SC-T-01, SC-T-02 |
| SC-C | 카테고리 관리 (Category) | SC-C-01, SC-C-02 |
| SC-F | 조회 및 필터링 (Filter) | SC-F-01, SC-F-02 |

## 4. 인증 시나리오 (SC-A)

---

### SC-A-01 — 신규 회원가입

**페르소나:** 페르소나 A (김지수, 32세, 마케터)

**배경 스토리:** 김지수는 업무·개인 할일을 동시에 관리하고 싶어 TodoListApp을 처음 접했다. 회사 PC에서 처음으로 회원가입을 시도한다.

**목표:** 이메일, 비밀번호, 이름을 입력하여 새 계정을 생성하고, 기본 카테고리가 자동 할당된 상태로 서비스를 시작한다.

**사전 조건:**
- 사용자가 회원가입 화면(SCR-01)에 접근한 상태
- 해당 이메일로 기존 계정이 존재하지 않음

**관련 FR/BR:** FR-A-01, BR-U-01, BR-U-02

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 이름("김지수"), 이메일("jisoo@example.com"), 비밀번호("Secure1234!"), 비밀번호 확인("Secure1234!")을 입력한다. | 각 필드 입력값의 형식을 실시간으로 검증하여 오류 없음을 표시한다. |
| 2 | 가입 버튼을 클릭한다. | 클라이언트에서 이메일 형식과 비밀번호 확인 일치 여부를 최종 검증한다. |
| 3 | (대기) | 서버로 회원가입 요청을 전송한다. 서버는 이메일 중복 여부를 검사한다. |
| 4 | (대기) | 서버가 비밀번호를 bcrypt(salt rounds 10 이상)로 해시하여 계정을 생성한다. |
| 5 | (대기) | UserRegistered 이벤트 발생: 기본 카테고리(일반, 업무, 개인) 3종을 해당 계정에 자동 연결한다. |
| 6 | (대기) | HTTP 201 응답 반환. 클라이언트는 "가입이 완료되었습니다." 메시지를 표시하고 로그인 화면(SCR-02)으로 이동한다. |

**결과:** 계정이 생성되었고, 기본 카테고리(일반, 업무, 개인)가 자동 연결된 상태다. 사용자는 로그인 화면에서 바로 로그인할 수 있다.

#### 예외 시나리오

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 이메일 형식 오류 | 이메일 필드에 "jisoo-example"을 입력하고 다른 필드로 포커스를 이동한다. | 클라이언트에서 즉시 "올바른 이메일 형식이 아닙니다." 오류 메시지를 표시하고 API 호출을 차단한다. | 가입 버튼이 비활성화되거나 버튼 클릭 시 서버 요청 없이 오류만 표시된다. |
| 비밀번호 확인 불일치 | 비밀번호("Secure1234!"), 비밀번호 확인("Secure5678!")을 각각 다르게 입력한 후 가입 버튼을 클릭한다. | 클라이언트에서 "비밀번호가 일치하지 않습니다." 오류 메시지를 표시하고 API 호출을 차단한다. | 비밀번호 확인 필드에 오류 상태가 유지되며 재입력을 유도한다. |
| 이메일 중복 | 이미 가입된 이메일("jisoo@example.com")로 가입 버튼을 클릭한다. | 서버가 중복 이메일을 감지하여 HTTP 409, "이미 사용 중인 이메일입니다." 메시지를 반환한다. | 이메일 필드에 오류 메시지가 표시되고 사용자는 다른 이메일을 입력해야 한다. |
| 필수 필드 누락 | 이름을 비워두고 가입 버튼을 클릭한다. | 서버가 HTTP 400, 누락된 필드(이름) 안내 메시지를 반환한다. (클라이언트 검증이 우선이나 서버도 동일하게 처리) | 해당 필드에 오류 메시지가 표시된다. |

### SC-A-02 — 이메일/비밀번호 로그인

**페르소나:** 페르소나 B (박현우, 28세, 개발자)

**배경 스토리:** 박현우는 출근 전 스마트폰으로 오늘 해야 할 사이드 프로젝트 할일을 확인하기 위해 앱에 로그인한다.

**목표:** 등록된 이메일과 비밀번호로 인증하여 JWT를 발급받고, 할일 목록 화면(SCR-03)으로 이동한다.

**사전 조건:**
- 사용자가 로그인 화면(SCR-02)에 접근한 상태
- 해당 이메일로 계정이 이미 존재함

**관련 FR/BR:** FR-A-02, BR-U-02

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 이메일("hyunwoo@example.com")과 비밀번호("MyPass5678!")를 입력한다. | 필드 입력 상태를 표시하며 별도 오류 없음. |
| 2 | 로그인 버튼을 클릭한다. | 클라이언트가 필수 필드 입력 여부를 확인하고 서버로 로그인 요청을 전송한다. |
| 3 | (대기) | 서버가 이메일로 사용자를 조회하고 입력된 비밀번호와 저장된 bcrypt 해시를 비교한다. |
| 4 | (대기) | 검증 성공 시 서버가 JWT(Access Token)를 생성하여 응답한다. UserLoggedIn 이벤트 발생. |
| 5 | (대기) | 클라이언트가 JWT를 Zustand store 메모리에 저장하고 인증 상태를 갱신한다. (localStorage 미사용) |
| 6 | (대기) | 할일 목록 화면(SCR-03)으로 자동 이동하며 자신의 할일 목록이 표시된다. |

**결과:** 인증된 상태로 할일 목록 화면에 진입하여 자신의 할일을 확인할 수 있다.

#### 예외 시나리오

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 이메일 미존재 또는 비밀번호 불일치 | 등록되지 않은 이메일이나 틀린 비밀번호를 입력하고 로그인 버튼을 클릭한다. | 서버가 HTTP 401, "이메일 또는 비밀번호가 올바르지 않습니다." 단일 메시지를 반환한다. (정보 노출 방지를 위해 원인을 구분하지 않음) | 로그인 폼에 오류 메시지가 표시되고 사용자는 다시 입력해야 한다. |
| 이메일 필드 누락 | 이메일을 비워두고 로그인 버튼을 클릭한다. | 서버가 HTTP 400, 누락된 필드 안내 메시지를 반환한다. (클라이언트 검증이 우선) | 이메일 필드에 오류 메시지가 표시된다. |
| 비밀번호 필드 누락 | 비밀번호를 비워두고 로그인 버튼을 클릭한다. | 서버가 HTTP 400, 누락된 필드 안내 메시지를 반환한다. | 비밀번호 필드에 오류 메시지가 표시된다. |

### SC-A-03 — 로그아웃

**페르소나:** 페르소나 A (김지수, 32세, 마케터)

**배경 스토리:** 김지수가 회사 PC에서 업무를 마치고 퇴근 전에 할일 앱에서 로그아웃한다. 공용 PC이므로 반드시 로그아웃이 필요하다.

**목표:** 현재 세션을 종료하고 로그인 화면으로 이동하여 다른 사용자가 내 데이터에 접근하지 못하도록 한다.

**사전 조건:**
- 사용자가 인증된 상태로 할일 목록 화면(SCR-03)에 있음

**관련 FR/BR:** FR-A-03

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 할일 목록 화면(SCR-03)에서 로그아웃 버튼을 클릭한다. | 로그아웃 처리를 시작한다. |
| 2 | (대기) | 클라이언트에서 저장된 JWT(Access Token)를 삭제한다. |
| 3 | (대기) | Zustand 전역 인증 상태를 초기화(사용자 정보, 토큰 등 모두 제거)한다. |
| 4 | (대기) | TanStack Query 캐시를 초기화하여 이전 사용자의 데이터가 잔존하지 않도록 한다. |
| 5 | (대기) | 로그인 화면(SCR-02)으로 자동 이동한다. |

**결과:** 인증 상태가 완전히 초기화되고 로그인 화면으로 이동한다. v1 정책상 서버 측 토큰 블랙리스트는 없으며, 클라이언트 토큰 삭제 방식으로 처리된다.

> **v1 보안 주의사항:** 로그아웃 후에도 토큰 만료 전까지는 탈취된 토큰이 이론적으로 유효할 수 있다. 이 한계는 v2 서버 측 블랙리스트 도입으로 해소할 예정이다.

#### 예외 시나리오

해당 없음. 로그아웃은 클라이언트 사이드에서 처리되므로 서버 오류에 의존하지 않는다.

## 5. 사용자 관리 시나리오 (SC-U)

---

### SC-U-01 — 개인정보 수정 (이름 및 비밀번호 변경)

**페르소나:** 페르소나 B (박현우, 28세, 개발자)

**배경 스토리:** 박현우는 보안을 위해 비밀번호를 주기적으로 변경하는 습관이 있다. 오늘 비밀번호를 바꾸면서 이름도 닉네임 형식으로 수정하려 한다.

**목표:** 이름을 변경하고 새 비밀번호로 업데이트한다.

**사전 조건:**
- 사용자가 인증된 상태로 개인정보 수정 화면(SCR-06)에 있음
- 유효한 JWT가 클라이언트에 존재함

**관련 FR/BR:** FR-U-01, BR-U-02, BR-U-04

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 개인정보 수정 화면(SCR-06)에 진입한다. | 현재 이름("박현우")을 이름 필드에 표시한다. 비밀번호 필드는 빈 상태로 표시한다. |

### SC-T-01 — 할일 등록

### SC-T-04 — 할일 완료 처리

**페르소나:** 페르소나 B (박현우, 28세, 개발자)

**배경 스토리:** 박현우는 오늘 아침 운동 루틴 30분을 마쳤다. 앱을 열어 "아침 운동 30분" 할일을 완료 처리하며 성취감을 느끼려 한다.

**목표:** 할일의 완료 여부를 미완료에서 완료 상태로 토글한다.

**사전 조건:**
- 사용자가 인증된 상태로 할일 목록 화면(SCR-03)에 있음
- 완료 처리 대상 할일의 isCompleted가 현재 false임

**관련 FR/BR:** FR-T-04, BR-T-04

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |

| 2 | (대기) | 서버로 완료 처리 요청을 비동기 전송한다. 서버가 JWT로 userId를 확인하고 소유권을 검증하여 isCompleted 값을 true로 반전(toggle)하여 저장한다. TodoCompleted 이벤트 발생. |

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 서버 오류 (5xx) | 체크박스를 클릭한다. | 낙관적 업데이트로 UI가 완료 상태로 변경되지만, 서버가 5xx 오류를 반환한다. | 클라이언트가 UI를 원래 상태(미완료)로 롤백하고 "잠시 후 다시 시도해 주세요." 오류 메시지를 표시한다. |
| 타인의 할일 완료 처리 시도 | (비정상적 접근) 타인의 할일 ID로 완료 처리 요청을 보낸다. | 서버가 HTTP 403을 반환한다. 낙관적 업데이트가 롤백된다. | UI가 원래 상태로 복원되고 오류 메시지가 표시된다. |

---

### SC-T-05 — 할일 단건 조회

**페르소나:** 모든 사용자

**배경 스토리:** 사용자가 할일 목록에서 특정 할일을 클릭하여 수정 화면으로 진입할 때, 서버에서 최신 데이터를 조회하여 폼에 정확히 pre-fill하는 과정이다.

**목표:** 수정 화면(SCR-04) 진입 전 서버에서 특정 할일의 최신 상세 데이터를 조회하여 폼에 pre-fill한다.

**사전 조건:**
- 사용자가 인증된 상태로 할일 목록 화면(SCR-03)에 있음
- 조회 대상 할일이 자신의 것임

**관련 FR/BR:** FR-T-05, BR-T-04, BR-U-03

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 할일 목록에서 특정 할일 카드(제목 영역)를 클릭한다. | 클라이언트가 해당 할일의 ID를 포함한 단건 조회 API를 호출한다. |
| 2 | (대기) | 서버가 JWT로 userId를 확인하고 소유권을 검증한다. |
| 3 | (대기) | 해당 할일의 전체 필드(제목, 설명, 종료예정일, 카테고리ID, 완료 여부)를 응답으로 반환한다. |
| 4 | (대기) | 할일 등록/수정 화면(SCR-04, 수정 모드)으로 이동하며 조회된 데이터가 폼의 각 필드에 pre-fill된다. |

**결과:** 수정 화면이 해당 할일의 최신 데이터로 채워진 상태로 표시된다.

#### 예외 시나리오

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 타인의 할일 조회 시도 | (비정상적 접근) 타인의 할일 ID로 단건 조회 API를 호출한다. | 서버가 HTTP 403, "접근 권한이 없습니다."를 반환한다. | 수정 화면으로 이동하지 않고 오류 메시지를 표시한다. |
| 존재하지 않는 할일 | 이미 삭제된 할일의 ID로 조회를 시도한다. (예: 다른 기기에서 삭제된 경우) | 서버가 HTTP 404, "할일을 찾을 수 없습니다."를 반환한다. | 수정 화면으로 이동하지 않고 "할일을 찾을 수 없습니다." 오류 메시지를 표시한다. 목록 화면 캐시를 갱신하여 해당 항목을 제거한다. |
| 토큰 만료 | 장시간 비활성 후 할일 카드를 클릭한다. | 서버가 HTTP 401을 반환한다. (BR-U-03) 클라이언트가 로그인 화면(SCR-02)으로 리다이렉트한다. | 단건 조회가 실패하고 재로그인이 필요하다. |

### SC-C-01 — 사용자 정의 카테고리 추가

**페르소나:** 페르소나 A (김지수, 32세, 마케터)

**배경 스토리:** 김지수는 기본 카테고리(일반, 업무, 개인) 외에 "사이드 프로젝트" 카테고리를 새로 만들어 부업 관련 할일을 별도로 관리하려 한다.

**목표:** 새 사용자 정의 카테고리 "사이드 프로젝트"를 생성한다.

**사전 조건:**
- 사용자가 인증된 상태로 카테고리 관리 화면(SCR-05)에 있음
- "사이드 프로젝트" 카테고리가 현재 존재하지 않음

**관련 FR/BR:** FR-C-01, BR-C-02, BR-C-03

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 카테고리 관리 화면(SCR-05)에 진입한다. | 기본 카테고리(일반, 업무, 개인)는 수정·삭제 불가 표시로 목록에 나타나고, 사용자 정의 카테고리 목록은 현재 비어 있다. |
| 2 | 카테고리 추가 입력 폼에 "사이드 프로젝트"를 입력한다. | 입력값이 필드에 표시된다. |
| 3 | 추가 버튼을 클릭한다. | 클라이언트에서 이름 입력 여부를 검증한다. 서버로 카테고리 생성 요청을 전송한다. |
| 4 | (대기) | 서버가 JWT로 userId를 확인하고 동일 사용자 범위 내 중복 이름 여부를 검사한다. (기본 카테고리명 포함) |
| 5 | (대기) | 중복 없음이 확인되면 userId를 자동 설정하고 isDefault = false로 카테고리를 생성한다. CategoryCreated 이벤트 발생. |
| 6 | (대기) | HTTP 201 응답 반환. 카테고리 목록이 갱신되어 "사이드 프로젝트"가 사용자 정의 카테고리 목록에 추가된다. 할일 등록/수정 화면의 카테고리 선택 목록도 갱신된다. |

**결과:** "사이드 프로젝트" 카테고리가 자신의 사용자 정의 카테고리로 생성되었다. 이후 할일 등록 시 이 카테고리를 선택할 수 있다.

#### 예외 시나리오

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 카테고리명 중복 (이미 동일한 사용자 정의 카테고리 존재) | 이미 자신이 만든 "사이드 프로젝트"를 또 추가하려 한다. | 서버가 HTTP 409, "이미 사용 중인 카테고리명입니다."를 반환한다. | 입력 필드에 오류 메시지가 표시된다. 다른 이름으로 재시도해야 한다. |
| 기본 카테고리명과 동일한 이름 입력 | "업무"라는 이름으로 카테고리 추가를 시도한다. | 서버가 HTTP 409, "이미 사용 중인 카테고리명입니다."를 반환한다. (기본 카테고리명도 중복으로 처리) | 입력 필드에 오류 메시지가 표시된다. |
| 카테고리명 미입력 | 이름 필드를 비운 채로 추가 버튼을 클릭한다. | 클라이언트에서 "카테고리명을 입력해 주세요." 오류 메시지를 표시하고 API 호출을 차단한다. | 서버 요청 없이 입력 필드에 오류 상태가 표시된다. |
| 토큰 만료 | 카테고리 추가 시도 시 이미 토큰이 만료된 상태다. | 서버가 HTTP 401을 반환한다. 클라이언트가 로그인 화면으로 리다이렉트한다. | 카테고리 생성이 처리되지 않는다. |

### SC-C-02 — 사용자 정의 카테고리 수정

**페르소나:** 페르소나 B (박현우, 28세, 개발자)

**배경 스토리:** 박현우는 이전에 만든 "독서" 카테고리를 "독서 & 학습"으로 이름을 더 명확하게 바꾸고 싶다.

**목표:** 기존 사용자 정의 카테고리 "독서"의 이름을 "독서 & 학습"으로 변경한다.

**사전 조건:**
- 사용자가 인증된 상태로 카테고리 관리 화면(SCR-05)에 있음
- "독서" 카테고리가 자신의 사용자 정의 카테고리로 존재함
- "독서 & 학습" 카테고리는 존재하지 않음

**관련 FR/BR:** FR-C-02, BR-C-01, BR-C-02, BR-C-03

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 카테고리 관리 화면(SCR-05)에서 "독서" 카테고리의 수정 버튼(또는 인라인 수정 영역)을 클릭한다. | "독서" 카테고리가 인라인 수정 상태로 전환된다. |
| 2 | 이름 필드의 "독서"를 지우고 "독서 & 학습"을 입력한다. | 수정된 이름이 필드에 표시된다. |
| 3 | 저장(확인) 버튼을 클릭한다. | 서버로 카테고리 수정 요청을 전송한다. |
| 4 | (대기) | 서버가 JWT로 userId를 확인하고 소유권을 검증한다. 중복 이름 여부를 검사한다. |
| 5 | (대기) | 검증 통과 시 카테고리명을 "독서 & 학습"으로 업데이트한다. CategoryUpdated 이벤트 발생. |
| 6 | (대기) | HTTP 200 응답 반환. 카테고리 목록이 갱신되어 "독서 & 학습"으로 표시된다. 해당 카테고리에 속한 할일 카드의 카테고리 표시도 갱신된다. |

**결과:** "독서" 카테고리가 "독서 & 학습"으로 변경되었다. 연결된 모든 할일의 카테고리 표시도 즉시 반영된다.

#### 예외 시나리오

| 타인의 카테고리 수정 시도 | (비정상적 접근) 타인의 카테고리 ID로 수정 요청을 보낸다. | 서버가 HTTP 403을 반환한다. | 수정이 거부된다. |

### SC-C-03 — 사용자 정의 카테고리 삭제

**페르소나:** 페르소나 C (이수현, 45세, 팀장)

**배경 스토리:** 이수현은 분기 프로젝트가 끝나 더 이상 사용하지 않는 "2Q 프로젝트" 카테고리를 정리하려 한다. 해당 카테고리에는 할일이 모두 삭제된 상태다.

**목표:** 할일이 없는 "2Q 프로젝트" 사용자 정의 카테고리를 삭제한다.

**사전 조건:**
- 사용자가 인증된 상태로 카테고리 관리 화면(SCR-05)에 있음
- "2Q 프로젝트" 카테고리가 자신의 사용자 정의 카테고리로 존재함
- "2Q 프로젝트" 카테고리에 연결된 할일이 없음

**관련 FR/BR:** FR-C-03, BR-C-01, BR-C-02, BR-C-04

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 카테고리 관리 화면(SCR-05)에서 "2Q 프로젝트" 카테고리의 삭제 버튼을 클릭한다. | 서버로 삭제 요청을 전송한다. |
| 2 | (대기) | 서버가 JWT로 userId를 확인하고 소유권을 검증한다. 해당 카테고리에 연결된 할일이 있는지 확인한다. |
| 3 | (대기) | 연결된 할일이 없음을 확인하고 카테고리를 삭제한다. CategoryDeleted 이벤트 발생. |
| 4 | (대기) | HTTP 200 응답 반환. 카테고리 목록에서 "2Q 프로젝트"가 제거된다. |

**결과:** "2Q 프로젝트" 카테고리가 삭제되었다. 할일 등록/수정 화면의 카테고리 선택 목록에서도 제거된다.

#### 예외 시나리오

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 기본 카테고리 삭제 시도 | 기본 카테고리 "일반"의 삭제 버튼을 클릭한다. | 삭제 버튼이 비활성화 상태이거나 서버가 HTTP 403, "기본 카테고리는 삭제할 수 없습니다."를 반환한다. | 삭제가 거부된다. |
| 연결된 할일이 있는 카테고리 삭제 시도 | 아직 할일이 남아 있는 "3Q 프로젝트" 카테고리의 삭제 버튼을 클릭한다. | 서버가 HTTP 409, "연결된 할일이 있어 삭제할 수 없습니다. 할일을 다른 카테고리로 이동하거나 먼저 삭제하세요."를 반환한다. | 카테고리가 삭제되지 않으며 오류 메시지가 표시된다. 사용자는 연결된 할일을 먼저 처리해야 한다. |
| 타인의 카테고리 삭제 시도 | (비정상적 접근) 타인의 카테고리 ID로 삭제 요청을 보낸다. | 서버가 HTTP 403을 반환한다. | 삭제가 거부된다. |

## 8. 조회 및 필터링 시나리오 (SC-F)

### SC-F-01 — 전체 할일 목록 조회

**페르소나:** 페르소나 C (이수현, 45세, 팀장)

**배경 스토리:** 이수현은 월요일 아침마다 앱을 열어 이번 주 처리해야 할 전체 할일을 한눈에 파악한다.

**목표:** 로그인 후 자신의 전체 할일 목록을 등록일 역순으로 확인한다.

**사전 조건:**
- 사용자가 인증된 상태임
- 하나 이상의 할일이 등록되어 있음

**관련 FR/BR:** FR-F-01, BR-T-04

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 로그인 성공 후 할일 목록 화면(SCR-03)에 자동 진입한다. | 클라이언트가 TanStack Query를 통해 할일 목록 API를 호출한다. |
| 2 | (대기) | 서버가 JWT로 userId를 확인하고 자신의 할일 전체를 조회한다. 기본 정렬은 등록일(createdAt) 내림차순이다. |
| 3 | (대기) | HTTP 200 응답과 함께 할일 목록 데이터를 반환한다. |
| 4 | 목록을 스크롤하며 확인한다. | 각 할일 카드에 제목, 카테고리, 종료예정일, 완료 여부 체크박스가 표시된다. |

**결과:** 등록된 모든 할일이 등록일 역순(최신 순)으로 표시된다. 다른 사용자의 할일은 절대 표시되지 않는다.

#### 예외 시나리오

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 등록된 할일 없음 | 신규 가입 직후 목록 화면에 진입한다. | 서버가 빈 배열 []을 반환한다. | "등록된 할일이 없습니다. 새 할일을 추가해 보세요." 빈 상태 메시지를 표시한다. |
| 비인증 상태로 목록 접근 | 토큰 없이 목록 API를 호출한다. | 서버가 HTTP 401을 반환한다. (BR-U-03) 클라이언트가 로그인 화면(SCR-02)으로 리다이렉트한다. | 할일 목록이 표시되지 않는다. |

### SC-F-02 — 카테고리별 필터링

**페르소나:** 페르소나 A (김지수, 32세, 마케터)

**배경 스토리:** 김지수는 현재 업무 관련 할일만 집중해서 보고 싶다. 카테고리 필터를 적용하여 "업무" 카테고리 할일만 표시한다.

**목표:** "업무" 카테고리를 선택하여 해당 카테고리의 할일만 목록에 표시한다.

**사전 조건:**
- 사용자가 인증된 상태로 할일 목록 화면(SCR-03)에 있음
- "업무" 카테고리에 할일이 하나 이상 존재함

**관련 FR/BR:** FR-F-02, BR-T-04

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 할일 목록 화면(SCR-03)의 카테고리 필터(탭 또는 드롭다운)에서 "업무"를 선택한다. | 클라이언트가 `categoryId=<업무_카테고리_ID>` 파라미터를 포함한 API를 호출한다. |
| 2 | (대기) | 서버가 JWT로 userId를 확인하고 "업무" 카테고리에 속한 할일만 필터링하여 반환한다. |
| 3 | 목록을 확인한다. | "업무" 카테고리의 할일만 등록일 역순으로 표시된다. 카테고리 필터 UI에 "업무" 선택 상태가 강조 표시된다. |
| 4 | 필터를 해제하기 위해 "전체" 탭을 선택한다. | 전체 할일 목록으로 복귀한다. |

**결과:** "업무" 카테고리에 속한 할일만 필터링되어 표시된다.

#### 예외 시나리오

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 선택 카테고리에 할일 없음 | 비어 있는 "사이드 프로젝트" 카테고리를 선택한다. | 서버가 빈 배열 []을 반환한다. | "해당하는 할일이 없습니다." 빈 상태 메시지를 표시한다. |

### SC-F-03 — 기간 필터링

**페르소나:** 페르소나 C (이수현, 45세, 팀장)

**배경 스토리:** 이수현은 이번 주(월~금)에 마감인 할일만 모아서 보고 싶다. 기간 필터로 이번 주 범위를 설정한다.

**목표:** 종료예정일이 이번 주 월요일~금요일 사이에 해당하는 할일만 필터링하여 확인한다.

**사전 조건:**
- 사용자가 인증된 상태로 할일 목록 화면(SCR-03)에 있음
- 이번 주 마감인 할일이 하나 이상 존재함

**관련 FR/BR:** FR-F-03, BR-T-04

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 할일 목록 화면(SCR-03)의 기간 필터에서 시작일을 이번 주 월요일로 설정한다. | 선택된 시작일이 필터 UI에 반영된다. |
| 2 | 종료일을 이번 주 금요일로 설정한다. | 선택된 종료일이 필터 UI에 반영된다. |
| 3 | 적용(조회) 버튼을 클릭한다. | 클라이언트가 `dueDateFrom=2026-05-11&dueDateTo=2026-05-15` 파라미터를 포함한 API를 호출한다. |
| 4 | (대기) | 서버가 종료예정일이 해당 기간에 속하는 할일만 필터링하여 반환한다. |
| 5 | 목록을 확인한다. | 이번 주 마감인 할일만 표시된다. |

**결과:** 종료예정일이 이번 주 범위에 해당하는 할일만 필터링되어 표시된다. 종료예정일이 없는 할일은 기간 필터 결과에서 제외된다.

#### 예외 시나리오

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 시작일이 종료일보다 늦은 경우 | 시작일을 2026-05-20, 종료일을 2026-05-10으로 설정한다. | 클라이언트에서 "시작일은 종료일보다 늦을 수 없습니다." 유효성 오류를 표시하고 API 호출을 차단한다. | 필터가 적용되지 않는다. 사용자가 날짜 범위를 수정해야 한다. |
| 유효하지 않은 날짜 형식 | 날짜 필드에 직접 잘못된 형식의 날짜를 입력한다. | 서버가 HTTP 400을 반환한다. | 오류 메시지가 표시된다. |
| 해당 기간에 할일 없음 | 먼 미래 날짜 범위로 필터링한다. | 서버가 빈 배열 []을 반환한다. | "해당하는 할일이 없습니다." 빈 상태 메시지를 표시한다. |

### SC-F-04 — 완료 여부 필터링

**페르소나:** 페르소나 B (박현우, 28세, 개발자)

**배경 스토리:** 박현우는 이번 달에 완료한 할일을 되돌아보며 성취감을 확인하려 한다. 완료 여부 필터로 완료된 항목만 모아서 본다.

**목표:** 완료 여부 필터를 "완료"로 설정하여 isCompleted = true인 할일만 표시한다.

**사전 조건:**
- 사용자가 인증된 상태로 할일 목록 화면(SCR-03)에 있음
- isCompleted = true인 할일이 하나 이상 존재함

**관련 FR/BR:** FR-F-04, BR-T-04

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 할일 목록 화면(SCR-03)의 완료 여부 필터 토글에서 "완료"를 선택한다. | 클라이언트가 `isCompleted=true` 파라미터를 포함한 API를 호출한다. |
| 2 | (대기) | 서버가 isCompleted = true인 할일만 필터링하여 반환한다. |
| 3 | 목록을 확인한다. | 완료된 할일만 표시된다. 완료 여부 필터 UI에 "완료" 선택 상태가 표시된다. |
| 4 | "미완료" 필터로 전환하기 위해 "미완료"를 선택한다. | `isCompleted=false` 파라미터로 API를 재호출하여 미완료 할일만 표시한다. |
| 5 | "전체"를 선택하여 필터를 해제한다. | 필터 파라미터 없이 전체 할일 목록을 다시 조회한다. |

**결과:** 완료된 할일만 목록에 표시되어 진척 상황을 확인할 수 있다.

#### 예외 시나리오

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 완료된 할일 없음 | "완료" 필터를 선택하지만 완료한 할일이 아직 없다. | 서버가 빈 배열 []을 반환한다. | "해당하는 할일이 없습니다." 빈 상태 메시지를 표시한다. |

### SC-F-05 — 복합 필터링

**페르소나:** 페르소나 A (김지수, 32세, 마케터)

**배경 스토리:** 김지수는 이번 주 마감인 업무 카테고리의 미완료 할일이 몇 개인지 빠르게 확인하려 한다. 카테고리, 기간, 완료 여부 세 가지 필터를 동시에 적용한다.

**목표:** 카테고리("업무"), 기간(이번 주), 완료 여부("미완료") 조건을 동시에 적용하여 조건에 맞는 할일만 표시한다.

**사전 조건:**
- 사용자가 인증된 상태로 할일 목록 화면(SCR-03)에 있음
- 다양한 카테고리, 종료예정일, 완료 여부를 가진 할일이 존재함

**관련 FR/BR:** FR-F-05, BR-T-04

#### 정상 시나리오

| 단계 | 사용자 행동 | 시스템 반응 |
|------|------------|------------|
| 1 | 카테고리 필터에서 "업무"를 선택한다. | 카테고리 필터 UI에 "업무" 선택 상태가 반영된다. |
| 2 | 기간 필터에서 시작일을 이번 주 월요일, 종료일을 이번 주 금요일로 설정한다. | 기간 필터 UI에 설정된 날짜가 반영된다. |
| 3 | 완료 여부 필터에서 "미완료"를 선택한다. | 완료 여부 필터 UI에 "미완료" 선택 상태가 반영된다. |
| 4 | 세 가지 필터 조건이 모두 설정되면 클라이언트가 자동으로(또는 조회 버튼 클릭 시) API를 호출한다. | `categoryId=<업무_ID>&dueDateFrom=2026-05-11&dueDateTo=2026-05-15&isCompleted=false` 파라미터로 API를 호출한다. |
| 5 | (대기) | 서버가 세 가지 조건을 모두 충족하는 할일만 필터링하여 반환한다. |
| 6 | 목록을 확인한다. | 이번 주 마감인 업무 카테고리의 미완료 할일만 표시된다. |

**결과:** 세 가지 조건을 동시에 충족하는 할일만 필터링되어 표시된다.

#### 예외 시나리오

| 예외 상황 | 사용자 행동 | 시스템 반응 | 결과 |
|-----------|------------|------------|------|
| 복합 조건에 해당하는 할일 없음 | 세 가지 필터를 설정했지만 조건을 모두 만족하는 할일이 없다. | 서버가 빈 배열 []을 반환한다. | "해당하는 할일이 없습니다." 빈 상태 메시지를 표시한다. 필터 조건을 조정하도록 안내한다. |
| 기간 필터에서 시작일이 종료일보다 늦은 경우 | 복합 필터 설정 중 기간을 잘못 입력한다. | 클라이언트에서 "시작일은 종료일보다 늦을 수 없습니다." 유효성 오류를 표시하고 API 호출을 차단한다. | 복합 필터가 적용되지 않는다. 기간 조건 수정 후 재시도해야 한다. |
