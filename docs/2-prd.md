## 2. 제품 개요

### 2.1 제품 비전

> "언제 어디서든 접속 가능한 나만의 할일 관리 공간 — 계정 기반으로 안전하게 보관되고, 카테고리로 체계적으로 정리된다."

### 2.2 제품 목표

| 목표 | 측정 기준 |
|------|-----------|
| 인증된 사용자의 할일을 안전하게 저장·관리한다 | 비인증 접근 차단율 100% |
| 카테고리 기반 분류로 할일 탐색 효율을 높인다 | 카테고리·필터 기능 완전 구현 |
| 복합 필터링으로 원하는 할일을 빠르게 찾는다 | 카테고리 + 기간 + 완료 여부 복합 조건 지원 |
| 반응형 UI로 데스크탑·모바일 모두 지원한다 | 주요 브레이크포인트(모바일/태블릿/데스크탑) 정상 렌더링 |

### 2.3 해결하는 문제

| 문제 | 해결 방안 |
|------|-----------|
| 개인별 할일을 체계적으로 관리할 수 있는 전용 앱 부재 | 사용자 인증 기반의 개인화된 할일 관리 기능 제공 |
| 인증된 사용자 단위로 데이터를 분리 저장하는 앱 부재 | 사용자 계정과 연동된 독립적인 데이터 저장 구조 적용 |
| 카테고리별로 할일을 분류·등록할 수 있는 앱 부재 | 기본 카테고리 및 사용자 정의 카테고리 등록 기능 제공 |

## 3. 타겟 사용자

### 3.1 주요 사용자 그룹

- **연령대:** 20대~50대 직장인
- **디바이스:** PC(업무 중), 스마트폰(이동 중·퇴근 후)
- **기술 수준:** 웹 서비스에 익숙한 일반 사용자 (개발 지식 불필요)

### 3.2 사용자 페르소나

---

**페르소나 A — 바쁜 직장인 김지수 (32세, 마케터)**

- **상황:** 여러 프로젝트의 업무·개인 일정을 동시에 관리해야 한다. 회사 PC와 퇴근길 스마트폰을 번갈아 사용한다.
- **목표:** 업무/개인 할일을 분리해서 관리하고, 마감일이 다가오는 항목을 한눈에 확인하고 싶다.
- **불만:** 메모 앱은 분류 기능이 없고, 협업 툴(Notion·Jira)은 개인 용도로 쓰기에 너무 복잡하다.
- **기대:** 로그인만 하면 바로 내 할일이 보이고, 카테고리·날짜 기준으로 빠르게 필터링할 수 있는 깔끔한 앱.

---

**페르소나 B — 자기계발 중인 직장인 박현우 (28세, 개발자)**

- **상황:** 업무 외 사이드 프로젝트, 독서, 운동 루틴 등 다양한 자기계발 목표를 추적한다.
- **목표:** 카테고리별로 할일을 구분하고, 완료된 항목을 한눈에 확인해 성취감을 얻고 싶다.
- **불만:** 기존 앱들은 계정 없이 쓰면 기기가 바뀔 때 데이터가 사라진다.
- **기대:** 계정 기반으로 어디서든 데이터가 동기화되고, 완료 여부 필터로 진척을 확인할 수 있는 앱.

---

**페르소나 C — 중간 관리자 이수현 (45세, 팀장)**

- **상황:** 팀 회의 준비, 보고서 제출, 개인 약속 등 다양한 종류의 할일이 뒤섞여 있다.
- **목표:** 종류별로 나눠서 관리하고, 기간 범위로 이번 주·이번 달 할일을 빠르게 추려보고 싶다.
- **불만:** 복잡한 UI는 배우는 데 시간이 걸려 부담스럽다.
- **기대:** 직관적인 UI, 필요한 기능만 딱 있는 심플한 앱.

---

### 3.3 사용자 공통 핵심 니즈 요약

| 니즈 | 대응 기능 |
|------|-----------|
| 기기 간 데이터 연속성 | 계정 기반 인증 + 서버 저장 |
| 할일 분류 및 탐색 | 카테고리 관리 + 복합 필터링 |
| 마감일 기반 우선순위 확인 | 종료예정일(dueDate) + 기간 필터 |
| 진척 상황 파악 | 완료 여부 토글 + 완료 필터 |
| 모바일·데스크탑 동시 사용 | 반응형 웹 UI |

- **지원 플랫폼:** Web + Mobile Web
- **구현 방식:** 반응형 웹 UI (네이티브 앱 없음)

- 중복 이메일 오류 메시지 표시

- 로그인 화면 이동 링크

#### SCR-02 — 로그인 화면

- 이메일, 비밀번호 입력 폼
- 자격증명 불일치 오류 메시지 표시
- 로그인 성공 시 할일 목록 화면으로 이동
- 회원가입 화면 이동 링크

---

#### SCR-03 — 할일 목록 화면 (메인)

- 내 할일 전체 목록 표시 (등록일 역순 기본 정렬)
- 카테고리 탭/드롭다운 필터
- 기간 필터 (시작일~종료일 날짜 선택)
- 완료 여부 토글 필터 (전체 / 미완료 / 완료)
- 복합 필터 동시 적용
- 할일 카드: 제목, 카테고리, 종료예정일, 완료 여부 표시
- 완료 여부 체크박스 (토글, UC-T-04)
- 할일 삭제 버튼 (UC-T-03)
- 할일 등록 버튼 → SCR-04 이동
- 할일 카드 클릭 → SCR-04 (수정 모드) 이동
- 카테고리 관리 버튼 → SCR-05 이동
- 개인정보 수정 버튼 → SCR-06 이동
- 로그아웃 버튼 (UC-A-03)

---

#### SCR-04 — 할일 등록/수정 화면

- 등록 모드 / 수정 모드 공용 화면
- 입력 필드: 제목(필수), 설명(선택), 종료예정일(선택), 카테고리(필수)
- 카테고리 선택: 기본 카테고리 + 사용자 정의 카테고리 드롭다운
- 수정 모드: 기존 값 pre-fill
- 저장 / 취소 버튼
- 제목 미입력·카테고리 미선택 시 오류 메시지 표시
- 저장 성공 시 할일 목록 화면으로 복귀

---

#### SCR-05 — 카테고리 관리 화면

- 기본 카테고리 목록 표시 (수정·삭제 불가 표시)
- 사용자 정의 카테고리 목록 표시 (수정·삭제 가능)
- 카테고리 추가 입력 폼 (이름 입력 + 추가 버튼)
- 카테고리명 인라인 수정
- 삭제 버튼 (연결 할일 존재 시 오류 메시지 표시)
- 중복 카테고리명 오류 메시지 표시

## 6. 기능 요구사항

### 6.1 인증 (Authentication)

---

#### FR-A-01 — 회원가입 (UC-A-01)

**목적:** 신규 사용자가 이메일, 비밀번호, 이름으로 계정을 생성한다.

**정상 흐름:**
1. 사용자가 이메일, 비밀번호, 이름을 입력하고 가입 버튼을 누른다.
2. 클라이언트에서 이메일 형식 및 비밀번호 확인 일치 여부를 검증한다.
3. 서버에서 이메일 중복 여부를 검사한다.
4. 비밀번호를 bcrypt로 해시하여 저장한다.
5. 계정 생성 후 기본 카테고리(일반, 업무, 개인)를 해당 사용자에게 자동 연결한다. (UserRegistered 이벤트)
6. 성공 응답을 반환하고 클라이언트는 로그인 화면으로 이동한다.

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 이메일 형식 오류 | 클라이언트 즉시 오류 표시, API 호출 차단 |
| 비밀번호 확인 불일치 | 클라이언트 즉시 오류 표시, API 호출 차단 |
| 이메일 중복 (BR-U-01) | HTTP 409, "이미 사용 중인 이메일입니다." |
| 필수 필드 누락 | HTTP 400, 누락 필드 안내 메시지 |

**비즈니스 규칙 연계:** BR-U-01, BR-U-02

---

#### FR-A-02 — 로그인 (UC-A-02)

**목적:** 등록된 사용자가 이메일/비밀번호로 인증하고 JWT를 발급받는다.

**정상 흐름:**
1. 사용자가 이메일, 비밀번호를 입력하고 로그인 버튼을 누른다.
2. 서버에서 이메일로 사용자를 조회한다.
3. 입력 비밀번호와 저장된 해시를 비교한다.
4. 검증 성공 시 JWT(Access Token)를 발급하여 응답한다.
5. 클라이언트는 JWT를 Zustand store 메모리에 저장하고 할일 목록 화면으로 이동한다.

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 이메일 미존재 또는 비밀번호 불일치 | HTTP 401, "이메일 또는 비밀번호가 올바르지 않습니다." (정보 노출 방지를 위해 단일 메시지) |
| 필수 필드 누락 | HTTP 400, 누락 필드 안내 메시지 |

**비즈니스 규칙 연계:** BR-U-02

---

#### FR-A-03 — 로그아웃 (UC-A-03)

**목적:** 인증된 사용자가 현재 세션을 종료한다.

**정상 흐름:**
1. 사용자가 로그아웃 버튼을 누른다.
2. 클라이언트에서 저장된 JWT를 삭제한다.
3. 클라이언트 상태(Zustand)를 초기화한다.
4. 로그인 화면으로 이동한다.

**예외 흐름:** 없음 (클라이언트 사이드 처리)

> **v1 정책 결정:** 도메인 정의서 UC-A-03은 "인증 토큰 무효화"로 정의되어 있으나, v1 개발 일정(3일) 제약상 서버 측 블랙리스트(Redis 등)를 구현하지 않고 클라이언트 토큰 삭제 방식으로 처리한다. 이에 따라 토큰이 만료되기 전 탈취된 경우 서버가 이를 무효화할 수 없다는 보안 한계가 존재하며, v2에서 서버 블랙리스트 방식으로 전환한다.

### 6.2 사용자 정보 관리 (User Profile)

---

#### FR-U-01 — 개인정보 수정 (UC-U-01)

**목적:** 인증된 사용자가 이름 및/또는 비밀번호를 변경한다.

**정상 흐름:**
1. 사용자가 수정할 항목(이름, 새 비밀번호)을 입력한다.
2. 클라이언트에서 비밀번호 확인 일치 여부를 검증한다.
3. 서버에서 JWT로 요청자를 식별한다.
4. 변경 항목을 검증 후 저장한다. 비밀번호 변경 시 bcrypt 재해시.
5. updatedAt을 갱신한다. (UserProfileUpdated 이벤트)
6. 성공 메시지를 표시한다.

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 비밀번호 확인 불일치 | 클라이언트 즉시 오류 표시 |
| 인증 토큰 없음/만료 | HTTP 401, 로그인 화면으로 리다이렉트 |
| 변경 항목 없음 | 저장 버튼 비활성화 또는 안내 메시지 |

**비즈니스 규칙 연계:** BR-U-02, BR-U-04

### 6.3 할일 관리 (Todo)

---

#### FR-T-01 — 할일 등록 (UC-T-01)

**목적:** 인증된 사용자가 새 할일을 생성한다.

**정상 흐름:**
1. 사용자가 제목(필수), 설명(선택), 종료예정일(선택), 카테고리(필수)를 입력한다.
2. 클라이언트에서 필수 필드 및 날짜 형식을 검증한다.
3. 서버에서 JWT로 userId를 확인한다.
4. isCompleted = false로 할일을 생성한다. (TodoCreated 이벤트)
5. 성공 응답 후 클라이언트는 목록을 갱신하고 목록 화면으로 복귀한다.

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 제목 누락 (BR-T-02) | 클라이언트 오류 표시, API 호출 차단 |
| 카테고리 미선택 (BR-T-03) | 클라이언트 오류 표시, API 호출 차단. 서버 도달 시 HTTP 400, "카테고리는 필수 항목입니다." |
| 유효하지 않은 날짜 형식 (BR-T-06) | HTTP 400, "유효하지 않은 날짜 형식입니다." |
| 인증 토큰 없음/만료 | HTTP 401 |

**비즈니스 규칙 연계:** BR-T-01, BR-T-02, BR-T-03, BR-T-05, BR-T-06

---

#### FR-T-02 — 할일 수정 (UC-T-02)

**목적:** 인증된 사용자가 자신의 할일을 수정한다.

**정상 흐름:**
1. 사용자가 수정할 할일을 선택한다.
2. 기존 값이 폼에 pre-fill된다.
3. 수정 후 저장 버튼을 누른다.
4. 서버에서 JWT로 userId를 확인하고 소유권을 검증한다.
5. 변경 내용을 저장하고 updatedAt을 갱신한다. (TodoUpdated 이벤트)

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 타인의 할일 수정 시도 (BR-T-04) | HTTP 403, "접근 권한이 없습니다." |
| 제목 삭제 후 저장 | HTTP 400, 제목 필수 오류 |
| 인증 토큰 없음/만료 | HTTP 401 |

**비즈니스 규칙 연계:** BR-T-02, BR-T-03, BR-T-04, BR-T-06

---

#### FR-T-03 — 할일 삭제 (UC-T-03)

**목적:** 인증된 사용자가 자신의 할일을 삭제한다.

**정상 흐름:**
1. 사용자가 삭제 버튼을 누른다.
2. 삭제 확인 다이얼로그를 표시한다.
3. 사용자가 확인하면 서버에 삭제 요청을 보낸다.
4. 서버에서 소유권을 검증 후 삭제한다. (TodoDeleted 이벤트)
5. 클라이언트 목록에서 해당 항목을 제거한다.

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 타인의 할일 삭제 시도 (BR-T-04) | HTTP 403 |
| 이미 삭제된 할일 | HTTP 404 |
| 인증 토큰 없음/만료 | HTTP 401 |

**비즈니스 규칙 연계:** BR-T-04

---

#### FR-T-04 — 완료 처리 (UC-T-04)

**목적:** 인증된 사용자가 할일의 완료 여부를 토글한다.

**정상 흐름:**
1. 사용자가 할일 카드의 체크박스를 클릭한다.
2. 서버에서 isCompleted 값을 반전(toggle)하여 저장한다. (TodoCompleted 이벤트)
3. 클라이언트에서 즉시 UI를 갱신한다. (낙관적 업데이트 권장)

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 타인의 할일 수정 시도 | HTTP 403 |
| 서버 오류 시 낙관적 업데이트 롤백 | UI 원상태 복원 + 오류 메시지 |

**비즈니스 규칙 연계:** BR-T-04

---

#### FR-T-05 — 할일 단건 조회

**목적:** 인증된 사용자가 수정 화면(SCR-04) 진입 시 특정 할일의 상세 데이터를 조회한다.

**정상 흐름:**
1. 사용자가 할일 목록에서 특정 할일 카드를 클릭한다.
2. 클라이언트가 해당 할일 ID로 단건 조회 API를 호출한다.
3. 서버에서 JWT로 userId를 확인하고 소유권을 검증한다.
4. 할일의 전체 필드(제목, 설명, 종료예정일, 카테고리, 완료 여부)를 반환한다.
5. SCR-04 폼에 기존 값이 pre-fill된다.

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 타인의 할일 조회 시도 (BR-T-04) | HTTP 403, "접근 권한이 없습니다." |
| 존재하지 않는 할일 | HTTP 404, "할일을 찾을 수 없습니다." |
| 인증 토큰 없음/만료 (BR-U-03) | HTTP 401, 로그인 화면으로 리다이렉트 |

**비즈니스 규칙 연계:** BR-T-04, BR-U-03

---

### 6.4 카테고리 관리 (Category)

---

#### FR-C-01 — 카테고리 추가 (UC-C-01)

**목적:** 인증된 사용자가 사용자 정의 카테고리를 생성한다.

**정상 흐름:**
1. 사용자가 카테고리명을 입력하고 추가 버튼을 누른다.
2. 서버에서 동일 사용자 범위 내 중복 여부를 검사한다.
3. 중복이 없으면 카테고리를 생성한다. (CategoryCreated 이벤트)
4. 카테고리 목록 및 할일 등록 폼의 카테고리 선택 목록을 갱신한다.

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 카테고리명 중복 (BR-C-03) | HTTP 409, "이미 사용 중인 카테고리명입니다." (기본 카테고리명과 동일한 이름도 중복으로 처리하여 생성 불가) |
| 카테고리명 미입력 | 클라이언트 오류 표시 |
| 인증 토큰 없음/만료 (BR-U-03) | HTTP 401 |

> **소유권 설정:** 생성 시 userId가 요청자의 id로 자동 설정되며 isDefault = false로 고정된다. 이후 해당 사용자만 수정·삭제할 수 있다. (BR-C-02)

**비즈니스 규칙 연계:** BR-C-02, BR-C-03

---

#### FR-C-02 — 카테고리 수정 (UC-C-02)

**목적:** 인증된 사용자가 자신의 사용자 정의 카테고리명을 변경한다.

**정상 흐름:**
1. 사용자가 수정할 사용자 정의 카테고리를 선택한다.
2. 새 카테고리명을 입력하고 저장한다.
3. 서버에서 소유권 확인 및 중복 검사 후 업데이트한다. (CategoryUpdated 이벤트)

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 기본 카테고리 수정 시도 (BR-C-01) | HTTP 403, "기본 카테고리는 수정할 수 없습니다." |
| 타인의 카테고리 수정 시도 (BR-C-02) | HTTP 403 |
| 중복 카테고리명 (BR-C-03) | HTTP 409 |

**비즈니스 규칙 연계:** BR-C-01, BR-C-02, BR-C-03

---

#### FR-C-03 — 카테고리 삭제 (UC-C-03)

**목적:** 인증된 사용자가 자신의 사용자 정의 카테고리를 삭제한다.

**정상 흐름:**
1. 사용자가 삭제할 사용자 정의 카테고리를 선택한다.
2. 서버에서 해당 카테고리에 연결된 할일이 있는지 확인한다.
3. 연결된 할일이 없으면 삭제한다. (CategoryDeleted 이벤트)
4. 카테고리 목록을 갱신한다.

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| 기본 카테고리 삭제 시도 (BR-C-01) | HTTP 403, "기본 카테고리는 삭제할 수 없습니다." |
| 연결된 할일 존재 (BR-C-04) | HTTP 409, "연결된 할일이 있어 삭제할 수 없습니다. 할일을 다른 카테고리로 이동하거나 먼저 삭제하세요." |
| 타인의 카테고리 삭제 시도 | HTTP 403 |

**비즈니스 규칙 연계:** BR-C-01, BR-C-02, BR-C-04

### 6.5 조회 및 필터링 (Todo List & Filtering)

---

#### FR-F-01 — 할일 목록 조회 (UC-F-01)

**목적:** 인증된 사용자가 자신의 전체 할일 목록을 조회한다.

**정상 흐름:**
1. 사용자가 메인 화면에 접속하면 JWT 기반으로 자신의 할일을 조회한다.
2. 기본 정렬: 등록일(createdAt) 내림차순.
3. 각 할일 카드에 제목, 카테고리, 종료예정일, 완료 여부를 표시한다.

**비즈니스 규칙 연계:** BR-T-04

---

#### FR-F-02 ~ FR-F-05 — 필터링 (UC-F-02~05)

**목적:** 카테고리, 기간, 완료 여부 조건을 단독 또는 복합으로 적용하여 할일 목록을 필터링한다.

| 필터 ID | 조건 | 파라미터 |
|---------|------|---------|
| FR-F-02 | 카테고리별 필터 | `categoryId` (단일 선택) |
| FR-F-03 | 종료예정일 기간 필터 | `dueDateFrom` (시작일), `dueDateTo` (종료일) |
| FR-F-04 | 완료 여부 필터 | `isCompleted` (true / false / 전체) |
| FR-F-05 | 복합 필터 | FR-F-02 ~ FR-F-04 조건 조합 |

**정상 흐름:**
1. 사용자가 필터 UI에서 조건을 선택한다.
2. 클라이언트에서 쿼리 파라미터를 구성하여 API를 호출한다.
3. 서버에서 조건에 맞는 할일 목록을 반환한다.
4. 조건에 해당하는 항목이 없을 경우 "해당하는 할일이 없습니다." 빈 상태 메시지를 표시한다.

**예외 흐름:**

| 예외 조건 | 처리 방식 |
|-----------|-----------|
| dueDateFrom > dueDateTo | 클라이언트 유효성 오류 표시 |
| 유효하지 않은 날짜 형식 | HTTP 400 |

## 7. 비기능 요구사항

### 7.1 성능

| 항목 | 목표 |
|------|------|
| 최대 동시 접속 사용자 | 300명 |
| 주요 API 응답 시간 (목록 조회) | 500ms 이하 (p95) |
| 페이지 초기 로딩 | 3초 이하 (LCP 기준, 일반 네트워크) |

### 7.2 보안

| 항목 | 요구사항 |
|------|---------|
| 비밀번호 저장 | bcrypt 해시 (salt rounds 최소 10) |
| 인증 방식 | JWT, 만료 시간 설정 필수 |
| 비인증 접근 차단 (BR-U-03) | 인증 토큰 없음/만료 시 HTTP 401 반환 후 로그인 화면 리다이렉트. 할일·카테고리·사용자 정보 관련 모든 엔드포인트에 JWT 검증 미들웨어를 횡단 적용한다. |
| API 인가 | 모든 인증 필요 엔드포인트에 JWT 검증 미들웨어 적용 |
| 데이터 격리 | 사용자는 자신의 데이터에만 접근 가능 (BR-T-04, BR-C-02) |
| HTTPS | 운영 환경에서 HTTPS 필수 |
| 입력 검증 | 서버 사이드 입력 검증 필수 (클라이언트 검증은 보조 수단) |
| SQL Injection 방지 | pg 라이브러리의 파라미터화된 쿼리(Parameterized Query) 사용 필수 |

### 7.3 가용성

| 항목 | 요구사항 |
|------|---------|
| 목표 가용성 | 소규모 개인 프로젝트 수준 (SLA 별도 정의 없음) |
| 데이터 보존 정책 | 회원 탈퇴 시 관련 데이터(Todo, 사용자 정의 카테고리) 즉시 삭제 |

### 7.4 확장성

| 항목 | 내용 |
|------|------|
| 인증 확장 | v2에서 OAuth 소셜 로그인 추가를 고려한 구조 설계 |
| 다크 모드 | v2에서 추가 예정, v1 CSS 구조에서 CSS 변수/토큰 방식 권장 |
| 다국어 지원 | v2에서 추가 예정, v1은 한국어 단일 |

### 7.5 유지보수성

| 항목 | 요구사항 |
|------|---------|
| TypeScript | 프론트엔드 전체 TypeScript 적용 |
| API 규격 | REST 컨벤션 준수 (HTTP 메서드, 상태 코드 표준 사용) |
| 환경 변수 | DB 접속 정보, JWT Secret 등 민감 정보는 환경 변수로 관리 |

## 8. 제약 사항 및 가정

| 제약 | 내용 |
|------|------|
| 개발 일정 | 3일 이내 MVP 완성. 범위 초과 기능은 v2로 이관. |
| 소규모 프로젝트 | 별도 인프라 모니터링, 로드 밸런서 구성 없음. |
| 다크 모드 | v1 미지원. |
| 다국어 | v1 한국어 단일 지원. |
| 탈퇴 기능 | v1 유스케이스에 미포함. 필요 시 v2에서 설계. |

### 8.3 가정

| 가정 | 내용 |
|------|------|
| 기본 카테고리 초기 데이터 | "일반", "업무", "개인" 3종을 DB 시드(seed) 데이터로 제공한다고 가정. |
| 페이지네이션 | v1에서는 전체 목록을 한 번에 로드한다고 가정 (사용자당 할일이 수백 건 미만). |
| 파일 첨부 없음 | 할일에 파일·이미지 첨부 기능은 없다고 가정. |
| 알림 기능 없음 | 종료예정일 알림(Push/Email) 기능은 없다고 가정. |

## 10. 개발 일정

| 시간대 | 작업 내용 | 산출물 |
